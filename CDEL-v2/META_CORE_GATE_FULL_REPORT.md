# Meta-Core Gate: In-Depth Report

## 1. Overview and Goals
This document builds on the executive summary captured in `CDEL-v2/REPORT_META_CORE_GATE.md:1-94`, where the Meta-Core gate work is described as a fail-closed overlay that covers every signed receipt flow in CDEL-v2 (sealed CCAI-X v1, sealed CCAI-X Mind v1/v2, repo-patch evaluation, and CAOE v1). The goal is to explain the new `cdel_meta_core_gate` package, how it hooks into the evaluation and receipt signing paths, how the CLI plumbing propagates the required `meta_core_root` argument, and how the unit and integration tests prove that the gate never issues a PASS receipt without signed meta-core provenance.

Because the meta-core root must now be provided to every receipt-producing command, we describe not only the Python modules that perform audit and injection but also the CLI flags, environment variable guards, and fixtures that keep the gate operational in long-running jobs and tests that do not have access to the full `meta-core` tree or the `agi-system` dependencies described elsewhere.

The canonical structure of the meta-core root is exposed through `tests/test_meta_core_gate_integration.py:1-94`, which copies the `engine`, `cli`, `kernel/verifier`, `meta_constitution/v1`, and `scripts/build.sh` directories from the real `meta-core` tree under `meta-core/`. That integration test symlinks or copies the necessary directories, stores the parent bundle, sets the active bundle hash, and executes the gate exactly as a sealed harness would. The reproduction demonstrates that the gate expects the CLI to live at `<meta-core-root>/cli/meta_core_audit_active.py` and that it reads engine/verifier artifacts downstream in order to compute the seven hash fields.

## 2. Meta-Core Gate Components
The new package under `cdel_meta_core_gate/` isolates every meta-core concern so that receipt writers only need to import a small surface area.

### 2.1 Domain Model
`cdel_meta_core_gate/domain.py:1-13` defines a frozen `MetaCoreAudit` dataclass with seven string fields (`active_bundle_hash`, `prev_active_bundle_hash`, `kernel_hash`, `meta_hash`, `ruleset_hash`, `toolchain_merkle_root`, `ledger_head_hash`). Once the gate successfully reads the CLI output, it instantiates this dataclass and passes it to the inject helper and any other consumer that needs to know which bundle was active. Having a dataclass centralizes the shape of the audit data and enforces that downstream code only reads named attributes, preventing ad-hoc dictionary lookups.

### 2.2 Audit Runner Implementation
`cdel_meta_core_gate/runner.py:39-123` is the most critical module. It enforces that the caller supplies an absolute path to the meta-core root, confirms that `<root>/cli/meta_core_audit_active.py` exists, and runs it under a tightly controlled environment (`PATH` plus `PYTHONUTF8=1`). The CLI executes with the meta-core root as an argument, writes JSON to a temporary file under the CDEL repository, and the runner enforces a 60-second timeout. The exit code handling is explicit: return code 2 becomes `MetaCoreGateInvalid`, while anything else (1, timeout, or a start failure) becomes `MetaCoreGateInternal`. The JSON payload is parsed, its keys are compared to the required set, and each hash field is validated with `_require_hash` or `_require_hash_or_empty`. This strict schema check prevents receipts from referencing partial or corrupted data. The runner returns a fully validated `MetaCoreAudit` instance, or it raises, meaning that no signature can ever be produced without a clean audit.

### 2.3 Receipt Injection Strategy
`cdel_meta_core_gate/inject.py:7-24` contains the deterministic injection helper. Receipt writers make a shallow copy of the body, add the `meta_core` object with data copied from the `MetaCoreAudit`, and return the new dict. The helper raises `MetaCoreGateInternal` if a receipt already has a `meta_core` entry, ensuring that the gate is only applied once and preventing late mutation. This pattern keeps all meta-core provenance attached to the receipt before canonical bytes are computed and before any signature operations run.

### 2.4 CLI and Error Surface
`cdel_meta_core_gate/errors.py:1-13` defines the base `MetaCoreGateError` plus `MetaCoreGateInvalid` and `MetaCoreGateInternal`, allowing the harnesses to treat invalid audits as evaluation failures (`CCAI_X_ERR_META_CORE_INVALID`, `CCAI_MIND_ERR_META_CORE_INVALID`) and internal audit issues as global aborts. `cdel_meta_core_gate/cli.py:24-44` provides the standalone gate CLI with the `audit_active` command. It simply calls `audit_meta_core_active`, prints the canonical JSON payload (with sorted keys for reproducibility), and exits with 0 / 2 / 1 depending on whether the gate succeeded, was invalid, or experienced an internal error. This CLI is the same entrypoint that the gate runner uses under the hood, so its behavior mirrors the requirements described in the `REPORT` summary.

### 2.5 Audit CLI Contract and Environment
Every call to the gate is scoped to an absolute root, and the JSON file it writes lives inside a temp directory under the CDEL repository, keeping the audit artifact located inside the evaluation workspace. The CLI arguments and environment are carefully scrubbed (`PATH` and `PYTHONUTF8` only), so the audit is reproducible even when evaluations run under tightly controlled sandboxes like the sealed worker. The runner’s strict validation ensures that audit data cannot be forged, tampered with, or bypassed, and the dataclass injection ensures the same JSON structure is used everywhere.

The internal `_REQUIRED_KEYS` constant (`cdel_meta_core_gate/runner.py:14-24`) enumerates the allowed JSON properties, so any future CLI change that adds or removes fields will cause the gate to reject the payload until the runner’s schema is updated. The temporary directory creation (`cdel_meta_core_gate/runner.py:53-90`) means audit artifacts are never left lying around after the process exits, which reduces attack surface if an evaluation fails. This also ensures layered CLI invocations (e.g., when other scripts shell out to `python3 -m cdel_meta_core_gate.cli`) can share the gate code without copying the same validation logic into multiple places.

## 3. Fail-Closed Guarantees Across Flows
All receipt flows now require a valid meta-core audit before signing. The sealed worker (`cdel/sealed/worker.py:40-123`) adds a `--meta-core-root` argument and falls back to `META_CORE_ROOT` as an environment variable when the CLI flag is absent. When evaluating CCAI-X v1 or Mind, the worker raises an immediate error if the root is missing, meaning the process never reaches the harness. Within the harnesses, the code keeps track of the environment variable (`meta_core_root_env`) and restores it once the run completes, ensuring other parts of the process remain undisturbed (`cdel/sealed/harnesses/ccai_x_v1/harness_v1.py:89-269` and `cdel/sealed/harnesses/ccai_x_mind_v1/harness_v1.py:72-427`).

If a gate call fails with `MetaCoreGateInvalid`, the harness changes the evaluation status from PASS to FAIL and records the new error code (e.g., `CCAI_X_ERR_META_CORE_INVALID` in `cdel/sealed/harnesses/ccai_x_v1/errors_v1.py:1-59` or `CCAI_MIND_ERR_META_CORE_INVALID` in `cdel/sealed/harnesses/ccai_x_mind_v1/errors_v1.py:1-60`). If the gate raises `MetaCoreGateInternal`, the harness aborts immediately, and no receipt is written. The repo-patch and CAOE flows rethrow the gate errors upward, so those commands also fail loudly when the audit path is compromised (`cdel/eval/repo_patch_eval_v1.py:74-232` and `extensions/caoe_v1/eval/run_eval_v1.py:536-714`). This layered fail-closed behavior mirrors the design described in the implementation report and ensures there is no inadvertent PASS without a signed meta-core audit.

The explicit mapping from `MetaCoreGateInvalid` to domain-specific fail codes also improves observability: when an audit returns invalid JSON or `verdict != "OK"` the sealed harness can log `CCAI_X_ERR_META_CORE_INVALID` or `CCAI_MIND_ERR_META_CORE_INVALID`, and automation can treat those failures differently from other evaluation failures. Internal gate failures bubble as exceptions, allowing supervision systems to treat them as environmental or infrastructure problems rather than candidate-specific issues. At the syscall level, the gate writes the audit output to `meta_core_audit.json` within a temporary directory inside the CDEL workspace, so logs can be inspected for debugging even when the gate aborts before emitting a receipt.

## 4. Receipt Integration Details
We now describe how the gate is threaded into each product path that issues a signed receipt.

### 4.1 Sealed CCAI-X v1
Receipt writers for CCAI-X v1 (`cdel/sealed/harnesses/ccai_x_v1/result_writer_v1.py:1-156`) call `audit_meta_core_active` during `build_receipt_payload`. After sorting suitepack receipts and computing evidence hashes, the function obtains the audit, injects it, and signs the body. The harness retains the `meta_core_root` from the worker, ensures the environment variable remains available while running sandbox checks, and re-injects it into the environment after evaluation (`cdel/sealed/harnesses/ccai_x_v1/harness_v1.py:72-269`). If the gate returns invalid, the harness sets `status = FAIL`, the fail code to `CCAI_X_ERR_META_CORE_INVALID`, and skips writing the receipt. If the gate cannot run (missing root or CLI), a `MetaCoreGateInternal` bubble-up causes the harness to raise `CcaiXError` with `CCAI_X_EVAL_INTERNAL`, aborting the job. The blanket attestation (`cdel/sealed/harnesses/ccai_x_v1/blanket_attest_v1.py:1-60`) removes `META_CORE_ROOT` before comparing the environment to the allowlist so the gate root is neither observed nor excluded; it is still recorded in the bootstrap list for audit purposes.

The evaluation writes `eval_result.json` with `write_eval_result` before trying to sign the receipt. That evaluation result always exists, even when the gate rejects the run; `write_eval_result` records the fail code in the JSON so a downstream consumer can distinguish meta-core failures from functional mistakes. Only once the run status is `PASS` does the harness call `build_receipt_payload`, which in turn calls `audit_meta_core_active`, injects the audit data, and runs `_sign_receipt_body` to produce Ed25519 signatures (`cdel/sealed/harnesses/ccai_x_v1/result_writer_v1.py:50-156`). The actual receipt writing process removes any previous `receipt.json`, so failures to sign (including meta-core rejections) guarantee the workspace either contains a PASS receipt with provenance or none at all.

### 4.2 Sealed CCAI-X Mind v1/v2
Mind receipts follow a similar injection path in `cdel/sealed/harnesses/ccai_x_mind_v1/result_writer_v1.py:1-158`, but they also support optional signing for certain integration scenarios. Regardless of how the receipt is built, meta-core audit data is inserted before canonical bytes are computed, and the function throws if the signing key is absent. The harness (`cdel/sealed/harnesses/ccai_x_mind_v1/harness_v1.py:72-427`) enforces `META_CORE_ROOT`, keeps track of plan and candidate metadata, and reruns the gate on every PASS attempt. When the gate returns invalid the harness uses `CCAI_MIND_ERR_META_CORE_INVALID`; when it fails internally, the evaluation stops altogether. The attestation helper (`cdel/sealed/harnesses/ccai_x_mind_v1/blanket_attest_v1.py:1-62`) is extended to remove not just `META_CORE_ROOT` but also several macOS-specific variables so allowlist checks remain deterministic even when meta-core root injection touches the environment.

The Mind harness also writes witness files, evidence bundles, and evaluation reports before the gate is run, which means meta-core failures are visible through multiple artifacts. `write_eval_result` captures the fail code, while the final receipt this helper tries to write includes the audit data and optional `signature_b64` field when signing keys are present (`cdel/sealed/harnesses/ccai_x_mind_v1/result_writer_v1.py:1-158`). Passing meta-core audits is the final gating step before the harness publishes `receipt.json`; if the gate rejects the audit, not only is the receipt abandoned, but witness files and evidence logs remain on disk so engineers can inspect the payload that failed.

### 4.3 Repo-Patch Evaluation
`cdel/eval/repo_patch_eval_v1.py:74-232` now requires a meta-core root before reading candidate bundles. The evaluation writes evidence and result reports as before but interposes the gate between building the receipt body and signing. When a PASS verdict is about to be signed, `audit_meta_core_active(meta_core_root)` is called, the metadata is injected, and the receipt is signed via `sign_receipt`. Any gate exception escapes the evaluation and prevents a PASS receipt, so manual or automated evaluation tools cannot skip the audit step. The CLI also exposes `--meta-core-root` (see Section 5) so repo-patch evaluators can pass the argument directly.

The `evidence_bundle` and `result_report` files are still constructed even when the gate rejects the audit. `_write_evidence` writes the bundle with information about the patch, policy, and plan, but `receipt` remains `None` until the gate succeeds and the canonical receipt body is updated with the meta-core hashes (`cdel/eval/repo_patch_eval_v1.py:200-230`). Because `render_envelope` returns `FAIL_BODY_BYTES` when the verdict is not PASS, there is no risk of leaking signed certificates containing stale or missing meta-core data.

### 4.4 CAOE v1 Verification
CAOE’s evaluation path (`extensions/caoe_v1/eval/run_eval_v1.py:536-714`) similarly enforces a meta-core root. The root is required; the function raises `MetaCoreGateInternal` when it is absent. The `_finalize` helper obtains the gate audit right before building the evidence report, ensures the `meta_core` object is part of the JSON, signs the report, and writes both `evidence_report.json` and `receipt.json`. The CAOE CLI (`extensions/caoe_v1/cli/caoe_cli_v1.py:12-64`) exposes `--meta-core-root` so users can supply the same path they use in sealed or repo-patch flows. CAOE also allows `META_CORE_ROOT` as an environment fallback for scripts.

The evaluation stores coverage and contract verdicts even when the gate rejects signing, but `_finalize` only calls `_build_evidence` and writes `receipt.json` when the audit succeeds. This means the meta-core gate is always the final gatekeeper for CAOE receipts: storage updates (evidence, lifecycle, progress) persist, yet only the audited receipt is disseminated. If the audit fails, operators can inspect the saved `coverage` metadata and `progress.json` files to understand why the gate never emitted a PASS certificate.

### 4.5 Common Attestation and Environment Handling
All evaluation paths share the same environment hypothesis: `META_CORE_ROOT` must either be provided through CLI or survive through fixtures. The sealed worker’s `meta_core_root_env` management restores the original environment after the run (`cdel/sealed/harnesses/ccai_x_v1/harness_v1.py:198-269` and `cdel/sealed/harnesses/ccai_x_mind_v1/harness_v1.py:131-427`). The tests’ `clear_env` helpers in `tests/ccai_x_v1/conftest.py:1-67` and `tests/ccai_x_mind_v1/test_ccai_x_mind_v1_harness.py:1-34` deliberately skip removing `META_CORE_ROOT`, ensuring the gate variables stay available during open-loop tests. Blanket attestations still log which bootstrap environment keys were removed, so auditors can see the gate’s footprint.

## 5. CLI and Operational Flow
The sealed worker now demands meta-core at startup (`cdel/sealed/worker.py:40-123`). The CLI’s `--meta-core-root` argument is passed to every harness invocation, guaranteeing consistent behavior across CCAI-X versions. `cdel/cli.py:73-132` adds the same argument to the `eval` command, which is vital for repo-patch evaluation where no sealed harness is involved. When the user runs `cdel eval --profile repo_patch_eval_v1 --candidate <bundle> --meta-core-root <ABS_PATH>`, the CLI extracts the argument and passes it straight into `repo_patch_eval_v1.evaluate_candidate_targ` via the evaluation service.

`extensions/caoe_v1/cli/caoe_cli_v1.py:12-64` adds `--meta-core-root` to `cdel caoe verify`, and the handler passes it to `evaluate_candidate`. This unifies the meta-core parameter across all entry points: sealed worker, high-level `cdel eval`, and CAOE verification share the same CLI argument name and environment fallback.

Separately, the gate provides its own CLI (`cdel_meta_core_gate/cli.py:24-44`). Operators can call `python3 -m cdel_meta_core_gate.cli audit_active --meta-core-root /abs/path/to/meta-core` to run the audit CLI in isolation, see the JSON output, and verify the same 64-character hashes that land inside receipts. The output is intentionally canonicalized, so scripts that pipe the output into other tools see deterministic JSON regardless of the hash ordering of Python dictionaries.

When running bundles in automation, the `meta_core_root` value should be considered a shared secret: it points to the audited meta-core repository that contains sensitive proof artifacts. The sealed worker’s argument (`cdel/sealed/worker.py:52-88`) and the `cdel` CLI argument group (`cdel/cli.py:73-132`) both accept absolute paths to avoid symlink attacks, and the gate enforces this requirement at runtime before executing the CLI (`cdel_meta_core_gate/runner.py:46-123`). In practice, this means `META_CORE_ROOT` should be set in the environment before launching any long-lived evaluation process so that all subprocesses inherit the same validated path.

## 6. Testing Strategy and Fixtures
The new tests cover both happy and unhappy paths for the audit gate (`tests/test_meta_core_gate_ok.py:1-45`, `tests/test_meta_core_gate_fail_closed.py:1-46`). The positive test writes a stub CLI that emits a valid JSON payload, runs `audit_meta_core_active`, and asserts that every field matches the payload. The negative test uses a stub CLI that either exits with code 2 or emits an invalid hash, ensuring `MetaCoreGateInvalid` is raised. `tests/test_meta_core_gate_receipt_injection.py:1-31` ensures injection duplicates the receipt instead of mutating it, and `tests/test_meta_core_gate_integration.py:1-94` walks a real meta-core repo fixture, runs `run_verify` from the actual meta-core engine, and checks the gate output with the real verification artifacts. This integration test is currently isolated from the main CI because it depends on the `meta-core` tree, but it documents how a full gate run behaves in practice.

The shared fixture in `tests/conftest.py:108-150` creates a fake meta-core root which contains a CLI script that prints deterministic hash values. The fixture sets `META_CORE_ROOT` for every test via an autouse fixture so that even non-meta-core tests see the gate root without needing to manage environment variables themselves. CAOE tests (`extensions/caoe_v1/tests/conftest.py:1-34`) and the CCAI-X harness tests re-use the same pattern, so developers can run evaluations in isolated workspaces with stub audits that return `verdict: OK`. The tests also disable the `benchmarks_cces/system_runtime` guard when those dependencies are missing, because `tests/test_meta_core_gate_integration.py` and similar rely on them.

`tests/test_meta_core_gate_integration.py:1-94` deserves special mention: it packages the real `meta-core` CLI, engine, and kernel verifier components into a temporary directory, runs `run_verify`, stores the resulting receipt and active bundle hash, and then replays the audit through `audit_meta_core_active` to ensure the hashes that land in the receipt match the meta-core output. Although it relies on the actual `meta-core` tree, the test proves the gate matches production expectations and serves as a reference implementation for any future gate consumers in other repositories.

## 7. Deployment and Usage Notes
Operational teams must always provide the meta-core root, either via the `--meta-core-root` CLI flag or by setting `META_CORE_ROOT` in the environment. The sealed worker explicitly checks for this in `cdel/sealed/worker.py:61-88`, and each harness raises a `MetaCoreGateInternal` if the root disappears partway through an evaluation. When running new evaluation scripts or automation, include the gate CLI earlier in the workflow, verify that `python3 -m cdel_meta_core_gate.cli audit_active` prints `verdict: OK`, and observe the seven `hash` keys to confirm the audit guard is functioning.

Adding the gate also means the environment sanitization logic in every evaluation flow must account for `META_CORE_ROOT`. The blanket attestation helpers were updated to temporarily remove this variable while logging that it was removed, so you cannot accidentally leak the gate root into the workload that is supposed to be hermetic.

In addition to gate-specific verification, ensure the rest of your evaluation harness still respects hermetic constraints: for CCAI we remove allowlisted env variables but keep `META_CORE_ROOT` out of the observable environment (`cdel/sealed/harnesses/ccai_x_v1/blanket_attest_v1.py:20-58`). When debugging, consult the `bootstrap_env_removed` list and inspect the temporary files under the run directory (`runs/.../receipt.json`) to confirm whether meta-core injection ever reached the signing step. If `receipt.json` is missing but `eval_result.json` has `CCAI_X_ERR_META_CORE_INVALID`, the gate rejected the audit before signing.

## 8. Appendix: Meta-Core Hash Fields
The audit data has seven fields, each 64-character hex; each one is validated by `_require_hash` or `_require_hash_or_empty` before a `MetaCoreAudit` instance is returned (`cdel_meta_core_gate/runner.py:103-123`):
1. `active_bundle_hash` (from `MetaCoreAudit`): the hash of the currently active bundle at the time of the audit; this is the fingerprint that gets recorded as the current state and appears inside every signed receipt.
2. `prev_active_bundle_hash`: the hash of the previous active bundle, which may be empty when no previous bundle exists, ensuring change detection across successive audits (`cdel_meta_core_gate/runner.py:103-123`).
3. `kernel_hash`: the hash of the kernel code that the meta-core root used for the audit, which binds the receipt to the interpreter implementation in use during verification.
4. `meta_hash`: the hash of the meta-constitution or metadata artifacts used for the audit, making the constitutional rules part of the provenance.
5. `ruleset_hash`: the hash of the rule machine executed during proof generation; this ensures the policy logic that governed the audit is recorded alongside the candidate evidence.
6. `toolchain_merkle_root`: the Merkle root of the toolchain that produced the audit, ensuring deterministic tool capture and preventing binary substitution attacks.
7. `ledger_head_hash`: the hash of the ledger head that was used to anchor the audit result, tying the receipt to the ledger snapshot against which the bundle was verified.

## 9. Observability and Troubleshooting
Gate failures are observable through returning non-zero exit codes and the new domain-specific error codes. `MetaCoreGateInvalid` maps to `CCAI_X_ERR_META_CORE_INVALID` and `CCAI_MIND_ERR_META_CORE_INVALID` by design (`cdel_meta_core_gate/errors.py:1-13`), while `MetaCoreGateInternal` causes the harness to abort and emit `CCAI_X_EVAL_INTERNAL` or `CCAI_MIND_EVAL_INTERNAL`. The standalone CLI mirrors this behavior; its exit codes of 0 / 2 / 1 let automation scripts know whether the audit succeeded, found an invalid payload, or experienced an internal issue (`cdel_meta_core_gate/cli.py:24-44`). `tests/test_meta_core_gate_fail_closed.py:1-46` covers both the invalid exit path (code 2) and schema errors so you can see how an invalid audit propagates to a raised exception before `inject_meta_core_fields` is even invoked.

The integration test `tests/test_meta_core_gate_integration.py:1-94` writes the actual `receipt.json` and asserts that the `meta_core` block matches the audit. When troubleshooting, inspect the temporary directory created in `_evaluate_candidate_tar` or the sealed harness run directory to see the `meta_core_audit.json` that the gate produced. That file is generated before the receipt writing logic (`cdel_meta_core_gate/runner.py:59-123`), so if it is missing, the gate never ran; if it exists but has `verdict != "OK"`, the gate intentionally raised `MetaCoreGateInvalid` and no signed receipt was written.

## 10. Suggested Next Steps
1. Add a CI job that runs `tests/test_meta_core_gate_integration.py` when the `meta-core` tree is synced, so the real gate CLI stays tested alongside the stubbed fixtures.
2. Document the gate CLI and environment contract in the public README so operators know to supply `--meta-core-root` to every receipt-producing command.
3. Monitor the meta-core CLI contract: if new audit fields are added, extend `_REQUIRED_KEYS` in `cdel_meta_core_gate/runner.py:14-24` accordingly and update the dataclass, injection helper, and receipts.

With these steps, the meta-core gate will remain a defensible, auditable, fail-closed layer across every CDEL receipt.
